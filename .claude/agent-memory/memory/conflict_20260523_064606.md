# Conflict resolution: re2-ed25519 <- remote-suspend

## Context
- Base: `ec683f04d18791d1e22f7e0388059d06de96dc6f`
- Ours: `re2-ed25519`, adding Ed25519-capable HTTP Signatures via `@misskey-dev/node-http-message-signatures`, additional local keys, and key selection plumbing.
- Theirs: updated `remote-suspend`, already containing `issue16270` and `ap-susp-update`, plus current `develop` changes such as Vitest/Rolldown migration, relay actor helpers, inbox hardening, and JSON-LD validation changes.

## Decisions
- `packages/backend/package.json` / `pnpm-lock.yaml`: used current develop dependency versions and added `@misskey-dev/node-http-message-signatures` from the PR branch, then regenerated the lockfile with pnpm.
- `RelayService`: kept develop's `getAcceptedRelays()` and `isRelayActor()` helpers while preserving the PR's optional `PrivateKeyWithPem` delivery path and `deliverMany` usage.
- `ApRequestService`: kept the PR's new signature library and local-key selection API. Adopted develop's GET behavior by not including `accept` in the signed header list.
- `InboxProcessorService`: kept the PR's `verifyDraftSignature` path and `getAuthUserFromApId(actorUri, keyId)` key resolution, while integrating develop's deleted-actor skip, old `acct:` key rejection, forbidden JSON-LD directive checks, JSON-LD freeze/error handling, and corrected federation block condition.
- `ActivityPubServerService`: kept the PR's `parseRequestSignature` / `verifyDigestHeader` path and retained develop's explicit Host check. Manual Digest parsing from the old library path was not kept because `verifyDigestHeader` covers the digest validation.
- Unit tests were resolved to current Vitest style, preserving PR-specific signature tests.

## Notes
- `UserSuspendService` and its tests were taken from updated `remote-suspend`; the final PR branch did not need to change their semantics.
- No merged migration was edited.
